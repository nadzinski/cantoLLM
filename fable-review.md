Everything checks out against the code, so here's my review.

## Big picture

The plan is in genuinely good shape, and unusually so for a solo learning project. The sequencing decisions are the ones I'd defend hardest: correctness-first einsum attention as a permanent reference implementation, batching before optimization so Phase 3 has a stable target, in-process scheduler before the IPC split, and the just-in-time refactor discipline. The Status-line convention works — I could reconstruct exactly where things stand from PLAN.md alone, which is the stated goal. The prototype-first approach to the scheduler (toy model, sequential oracle, token-for-token equivalence tests) is exactly how I'd de-risk the hand-written version, and the prototype itself is clean: the water-fill budget allocator is a nice touch, and I verified the guard `max_tokens_per_step >= max_batch` does guarantee every active sequence gets ≥1 token per step, so no zero-width rows.

The new design note (`continuous_batching.md`) is accurate — I checked its claims about the scalar `start_pos`, the 2D mask protocol, and the `InferenceBackend.generate(seq)` mismatch against the code, and they all hold. Its recommendation to bypass `InferenceBackend` for the batched engine and break at the `AttentionMethod` seam is the right call. On its open question #1 (cache shape): I'd commit to the runtime-owned pool with `Sequence` carrying a slot handle, because Phase 4's paged cache is the same shape — a block table is also just a handle into a runtime-owned pool, so you'd only design the seam once. Note the file is still untracked; PLAN.md's "design note in-repo" commitment suggests committing it (maybe under a `notes/` directory).

**The one plan-level risk I'd flag: the bench harness spec has been "Open" since Phase 0 (2026-04-19), but Phase 2's exit explicitly depends on it** — "end-of-phase numbers become the 'before' baseline that Phase 3's CUDA optimizations measure against." If the spec keeps slipping, you'll finish the scheduler with nothing to measure it with and Phase 3 loses its before/after story. I'd time-box the spec before the scheduler work finishes, not after.

## Bugs

**1. `SequentialEngine` doesn't actually enforce sequentiality — and `SpeculativeBackend` has shared mutable state. (Real, today.)** `submit()` (`src/cantollm/engine/sequential.py:41`) spawns a worker thread per request with no lock or queue, so two concurrent HTTP requests run `backend.generate()` concurrently. For `StandardBackend` that's accidentally safe (each `Sequence` has its own cache, the model is stateless). But `SpeculativeBackend` owns a single `self.draft_cache` and stats counters (`speculative.py:27-29`), and `run()` calls `backend.reset()` per request — so request B clears request A's in-flight draft cache mid-generation and both produce garbage. There's no admission control yet (deferred to Phase 3.5), so nothing upstream prevents this. A one-line `threading.Lock` around the generate path (or an asyncio semaphore in `submit`) would make the engine match its name until 3.5 lands.

**2. Prototype scheduler has no `max_seq_len` enforcement — one overlong request crashes the step for everyone.** Nothing checks `position + num_new <= cache.max_seq_len`, so a prompt (or prompt+generation) exceeding the slot length hits a shape-mismatch error on the slice assignment in `toy_model.py:91`, and since the whole batch shares one forward call, every active sequence dies with it. The design note's point 9 discusses pool *sizing* but not *enforcement* — at integration you need both an admission check (`prompt_len + max_tokens ≤ cap` → reject at the API) and a per-step guard. Fine to leave out of the prototype, but it should be on the integration checklist explicitly.

**3. Stop-token contract diverges between prototype and real backend — this will bite the equivalence tests.** The prototype emits the stop token then finishes (`scheduler.py:145`, matching `reference.py`), but `StandardBackend.generate` returns *without* yielding it (`standard.py:103`). The real behavior is the correct one for an API (clients shouldn't see `<|im_end|>`). When you port the scheduler, the stop check has to move *before* emission, and the planned "batched-vs-sequential token-for-token" tests against tiny-Qwen3 will diverge on the final token until it does. Related: the prototype's finish event carries both `token_id` and `finish_reason` on one `TokenEvent`, while the real contract (`engine/types.py:84`) says exactly one field is populated — the multiplexer will need the two-event shape.

**4. The greedy sampling shortcut is a time bomb given your own roadmap.** `StandardBackend.sample` (`standard.py:52`) skips the processor pipeline when `greedy=True`, justified as "argmax is invariant under monotonic shifts." True for temperature/top-p — false for repetition penalty, logit bias, and guided decoding, all of which PLAN.md explicitly names as future pipeline extensions. The first one that lands will silently not apply at temperature 0. Worth a comment-level tripwire now, or making `from_temperature_top_p` the only place allowed to set `greedy=True`.

**5. Two small prototype nits.** `_finish_reason` uses `==` against `max_tokens` (`scheduler.py:148`) — with `max_tokens=0` the count skips past and the request generates forever; use `>=`. And `run_with_late_arrivals` (`runner.py:58`) breaks on idle *before* future scheduled arrivals are admitted, so a request scheduled after an idle gap is silently dropped — a latent trap for future tests rather than a current failure.

**6. Design-note arithmetic check.** Point 9 claims ~6 GB KV for 0.6B at `max_batch=8, max_seq_len=4096`. I get ~3.8 GB: 2 (K+V) × 28 layers × 8 KV heads × 128 head_dim × 2 bytes ≈ 112 KB/token, × 32768 slot-tokens ≈ 3.8 GB. The conclusion (need a separate batched-cache cap; two max-seq-lens coexist in config) stands either way, but worth re-deriving before it drives a config default.

## Smaller observations

- `PaddedKVCache.allocate_slot` pops from a `set`, so slot assignment order is arbitrary. Output correctness doesn't depend on it, but deterministic slot assignment (e.g., a sorted free list or `deque`) makes failures reproducible when debugging the real version — cheap insurance.
- The scheduler samples and discards a token for every non-final prefill chunk. Harmless in the prototype, and on the real model the lm_head over `(B, vocab)` is cheap relative to the forward, but the integrated version can skip the lm_head for rows still prefilling — worth remembering when perf measurement starts in Phase 3.
- `SequentialEngine`'s backpressure/drain dance around the bounded queue (`sequential.py:93-112`) is correct as far as I can trace it — the disconnect path sets `stop_event`, drains so a blocked `put` unblocks, and awaits the worker. Good groundwork for the trickier multiplexed version the design note's point 7 describes.

Overall: the project is healthy, the plan is the strongest artifact in it, and the prototype is ready to integrate. The two things I'd act on before scheduler integration are the `SequentialEngine`/speculative concurrency hole (real bug, reachable today) and pinning down the stop-token emission contract, since the equivalence tests you're about to lean on depend on it.
